# ADR-001: Use Kafka as Central Transport for Payment Service Observability Pipeline

## Status

Accepted (2026-06-03)

---

## Context

We are building an AIOps anomaly detection platform for a **payment service mesh** consisting of 10–50 microservices (auth, fraud-detector, wallet, gateway, notification, reconciliation, etc.). Each service emits:

- **Metrics**: ~100K–500K data points/sec (transaction rate, latency histograms, error counters)
- **Logs**: ~50–200 GB/day (structured JSON, severity-tagged)
- **Traces**: ~5K–50K spans/sec (distributed transaction traces)

### Problem

The initial architecture used **direct push**: each OTel Collector exporter sends data simultaneously to VictoriaMetrics, Loki, and Tempo. At peak load (Black Friday, month-end settlement), this caused:

- **~8% event drop rate** in VictoriaMetrics at 300K events/sec due to write backpressure
- **Loki ingest lag** up to 45 seconds, making real-time alerting unreliable
- **No replay capability**: if the ML feature extraction pipeline crashes, all in-flight data is lost
- **Fan-out coupling**: adding a new consumer (e.g., Spark batch training pipeline) requires modifying every OTel Collector config

We evaluated four transport options:

| Option | Throughput | Replay | Decoupling | Cost/month |
|--------|-----------|--------|------------|------------|
| Direct push (current) | 200K/sec before drop | ✗ No | ✗ Tight | $0 extra |
| Direct push + rate limit | 200K/sec capped | ✗ No | ✗ Tight | $0 extra |
| Vector aggregator | 400K/sec | ✗ No (in-memory buffer only) | Partial | ~$80 |
| **Apache Kafka (chosen)** | **1M+/sec** | **✓ 7-day retention** | **✓ Full** | **~$200–600** |
| Confluent Cloud serverless | 1M+/sec | ✓ 7-day | ✓ Full | ~$150–400 |

---

## Decision

**Introduce Apache Kafka (or Confluent Cloud Serverless for personal/startup accounts) as the central transport layer** between OTel Collectors and all downstream consumers (VictoriaMetrics, Loki, Flink feature pipeline, Spark batch trainer).

Architecture:

```
OTel Collector → Kafka topics → { VictoriaMetrics (via Kafka connector)
                                 { Loki (via Promtail Kafka reader)
                                 { Flink (feature extraction + anomaly scoring)
                                 { Spark (nightly model retraining)
```

Three Kafka topics:
- `payment.metrics` — time-series data points, 200-byte avg message
- `payment.logs` — structured log lines, 500-byte avg message
- `payment.traces` — span data, 1KB avg message, 1% sampling

Configuration:
- Replication factor: 3
- Retention: 7 days (enables ML pipeline replay + incident replay)
- Partitions: 12 per topic (matches Flink parallelism)

---

## Consequences

### Positive

- **Zero event drop**: Kafka absorbs burst writes; downstream systems consume at their own pace. Backpressure handled at consumer level, not producer.
- **Replay capability**: If Flink anomaly scoring job crashes, it can rewind the offset and reprocess the last 7 days — critical for payment auditability.
- **Decoupling**: Adding a new consumer (e.g., a SIEM system, a new ML model) requires zero changes to services or OTel Collectors.
- **Multiple independent consumers**: Flink (real-time, <2s latency) and Spark (batch, nightly) can consume the same `payment.metrics` topic independently.
- **Throughput headroom**: Kafka sustains 1M+ messages/sec on a 3-broker cluster; well above our 500K peak.
- **End-to-end latency impact**: +8–12ms median added to telemetry pipeline (acceptable — alerting SLO is 30s, not sub-second).

### Negative

- **Operational complexity**: Kafka requires broker health monitoring, partition rebalancing, offset management, JVM tuning. Estimated 0.2–0.3 FTE SRE time/month.
- **Added cost**: $200–600/month for self-hosted Kafka (3× `e2-standard-4` on GCP). Confluent Cloud Serverless is $150–400/month at our throughput, with lower ops burden.
- **Not suitable for real-time alerting <5s**: If alert latency requirement is <2s end-to-end, Kafka's consumer polling interval (500ms default) adds jitter. Mitigation: use Kafka with `fetch.min.bytes=1` + direct OTel→Alertmanager path for critical alerts only.
- **Storage cost for replay**: 7-day retention at 500GB log/day ≈ 3.5TB Kafka storage ≈ $140/month on GCS-backed Kafka (Tiered Storage).

### Cost Summary

| Item | Before (direct push) | After (with Kafka) | Delta |
|------|--------------------|-------------------|-------|
| Event drop rate | ~8% (at 300K/sec) | ~0% | −8% |
| Storage (re-processing wasted work) | $120/month | $0 | −$120 |
| Kafka cluster (self-hosted) | $0 | +$400/month | +$400 |
| SRE ops time | 0.1 FTE | +0.25 FTE (~$1,200/month @ $80K/yr) | +$1,200 |
| **Net monthly cost change** | | | **+$1,480** |
| **Value: recovered 8% of $2M/day txn data** | | **~$160K/month risk avoided** | **✓ justified** |

## Alternatives Considered

### 1. Scale direct push horizontally
Add more VictoriaMetrics write nodes to absorb 300K+ events/sec.

- Cost: $50K+/month for 3× throughput at peak
- No replay capability
- Tight coupling remains
- **Rejected**: too expensive and does not solve replay/decoupling

### 2. Direct push with rate limiting (OTel tail sampling + head-based drop)
Cap at 200K events/sec via OTel Collector `probabilistic_sampler`.

- Cost: $0 extra
- Systematic data loss — **unacceptable for payment fraud detection** where every transaction must be observable
- **Rejected**: violates data completeness requirement

### 3. Vector aggregator (Vector.dev) without Kafka
Vector provides in-memory buffering + disk spill (~4GB buffer).

- Cost: +$80/month (1 VM node)
- Throughput: ~400K events/sec per node
- No replay (disk buffer, not log-structured)
- No multi-consumer fan-out
- **Rejected**: insufficient replay capability; adds complexity without solving fan-out problem

### 4. Confluent Cloud Serverless (fully managed Kafka)
Same as decision but fully managed.

- Cost: $150–400/month (vs $400–600 self-hosted)
- Zero ops overhead
- **Recommended for personal/startup accounts** (Series A, <50 services): lower TCO when SRE time is scarce
- Can migrate to self-hosted later when team scales past 3 SREs
